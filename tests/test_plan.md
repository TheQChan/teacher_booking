# План тестирования Teacher Booking

## Цель
Обеспечить, чтобы ключевые backend‑флоу (авторизация, роли, расписание и регистрация) продолжали работать независимо от изменений приложения, а документация тестов оставалась в одном месте.

## Область и задачи
1. **Аутентификация и профиль** — проверить регистрацию, уникальность, подтверждение пароля и публичный `/api/v1/drf-auth/user/`.
2. **Сервис расписаний** — гарантировать, что только учителя создают слоты, запрещается наложение, своевременно проверяется порядок `start`/`end`, а действия `schedule`/`free`/`cancel`/`complete` защищены.
3. **Ограждения бизнес процессов** — убедиться, что студент может только записаться/освободить свои слоты, а учитель отменяет/завершает записи, и что выражения ошибок очевидны (HTTP-коды, текст).
4. **Готовность фронтенда** — API возвращает то, что ожидает SPA; любые регрессии ломают тесты до доставки на UI.

## Стратегия
- `api/` содержит Django `TestCase` с `APIClient`, но запускается через `pytest`, чтобы перейти на современный runner.
- Каждое тестовое действие описано в `test_cases/` для согласованности с документацией и будущими BDD/GUI-скриптами.
- Силу сценариев можно расширить BDD/GUI/нагрузочными наборами, разместив новые фреймворки в `bdd/`, `gui/` и `load/`.

## Окружение
- Python 3.12+ (как у основного проекта).  
- Зависимости в `requirements.txt` (Django/DRF + `pytest`/`pytest-django`). BDD/GU/etc могут добавлять свои `requirements` рядом с соответствующими каталогами.
- Бекенд должен быть доступен по пути `../backend/teacher_booking`, чтобы `conftest.py` мог подключить настройки и модели.

## Критерии входа/выхода
- **Вход**: схемы/миграции применены, чтобы модель `Session` и `UserInfo` совпадали с тестами.  
- **Выход**: `pytest` проходит за приемлемое время (<60 с локально) и/или генерирует отчёт покрытия (`coverage run -m pytest`).

## Риски и смягчения
- **Дрейф API**: тесты завязаны на конкретные пути и статусы; при смене API обновить `api/` и пояснения в `test_cases/`.
- **Зависимость от структуры**: `conftest.py` ожидает `../backend/teacher_booking`; не меняйте расположение папок, или обновите конфиг.
- **Будущие фреймворки**: BDD/GUI/нагрузка может требовать собственного окружения — указывайте инструкции внутри `bdd/README.md`, `gui/README.md`, `load/README.md`.

## Текущая активность
- Пополнять `test_cases/` новыми сценариями при добавлении тестов в `api/`.  
- Внедрять BDD/GUI/нагрузочные проверки как отдельные подпроекты (режим `pytest`, Selenium, locust/k6 и т.п.).  
- Запускать `pytest` регулярно в локальной среде и CI, убедившись, что путь к бекенду доступен.
